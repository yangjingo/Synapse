# BLOG-STYLE-CN: Why.J 研究与工程写作元审美

## 0. 活人感（不可妥协）

**Blog 不是 DSL，不是 Slide。Blog 是人写的散文，写给活人看的。**

DSL 标签（`::pulse`、`::math`、`::code`）**禁止**出现在 Blog 输出中。它们属于幻灯片，不属于段落。

### 什么是"活人感"

- **像凌晨两点给同事解释问题那样写**，不像下午两点写论文那样写。
- **口语可以，粗糙可以，草稿里的错别字可以。** 过度润色消灭的是人味。
- **个人判断保持个人化**："Python 是我最熟悉的工具" > "生态轻量、SDK 成熟"。读者来的是 *你的* 观点，不是维基百科摘要。
- **情绪是信号，不是噪音**："跑不起来，没有手感" 传达的信息量远大于 "缺乏运行环境的静态阅读无法感知动态反馈"。
- **兴奋带来的啰嗦是好的**：8 条发自真实的反思，好过压缩后的 5 条"Action Items"。

### 润色陷阱

润色草稿时，以下转换会**消灭**人味：

| 原文（保留） | 过度润色（避免） |
|-------------|-----------------|
| "我只是在用，并不理解它。" | "陷入'使用幻觉'的认知熵增" |
| "牵出十几个模块" | "依赖纠缠" |
| "删掉之后系统还能跑吗？" | "每一层删除都是一次深度考问" |
| "觉得好玩就开始" | （被整条删除） |
| "Python 是我最熟悉的" | "能最大程度降低非核心细节的干扰" |

**原则**：如果润色后这句话像是任何人都能写出来的，那就是错误的润色。如果听起来还是 *你*，就保留。

## 1. 写作 DNA

### OpenAI 风格 (数据驱动)

- **结构**: 挑战 → 解决方案 → 指标。
- **语调**: 极简、专业、高权威感。

### Anthropic 风格 (严谨)

- **结构**: 问题 → 详细方法 → 局限性 → 讨论。
- **语调**: 深度解释、对话式但保持学术感。

### Karpathy 风格 (直觉)

- **结构**: 钩子 (Hook) → 分步“配方” (Recipe) → 提示与技巧。
- **语调**: 强个人风格、“发生了什么”的代码块。

## 3. Synapse 创作五律 (The Synapse Ethos)

这是一种典型的**“硬核工程师随笔”**风格。它是写给那些“深夜还在调参数、看日志、对满天飞的新名词冷冷一笑”的技术同类看的。

### I. 祛魅原则 (Anti-Hype / No-Bullshit)
- **核心：** 拒绝名词泡沫，直击工程本质。
- **执行：** 遇到高大上的新词（如 Agentic Workflow），自动转译为具体的工程实现（如“带状态机的循环逻辑”）。不卖关子，不讲宏大叙事，首段即进入战场。

### II. 伤痕美学 (Failure-Driven / Practical Realism)
- **核心：** 失败的细节比成功的结论更有信息熵。
- **执行：** 详细记录“因为某某限制导致某某方案由于某某原因失败”的过程。用“坑”来定义“路”，让内容带有真实的粗糙感和现场感。

### III. 古典基座 (Foundational Logic / Classic Modernism)
- **核心：** 前沿 AI 问题的解药往往藏在经典软件工程里。
- **执行：** 强调解耦、约束、确定性。在讨论神经网络或大模型时，始终保持架构师的视角，寻找与单一职责（SRP）或关注点分离（SoC）的映射关系。

### IV. 机械精度 (Terminological Precision / Geek Rationalism)
- **核心：** 词语是思想的刻度。
- **执行：** 使用具有机械感和结构感的动词（如：注入、暴露、编排、路由、原子化）。排版推崇 Markdown 的原生结构，追求高信息密度。

### V. 智性克制 (Skeptical Humility)
- **核心：** 对技术奇点保持警惕，对当下实践保持敬畏。
- **执行：** 结论不给定论，多用设问或开放式结尾。承认技术的局限性，比吹嘘技术的全能更显专业。

## 5. 句式红黑榜 (Sentence Pattern Audit)

### A. 禁区：油腻营销号句式（生理性厌恶级）

**特点**：好为人师、辞藻空洞、强行升华、二元对立。**任何一条出现，整段重写。**

| 句式类型 | 典型模版 | 为什么油腻？ |
| :--- | :--- | :--- |
| **虚假的深刻** | "不是 **XXX**，而是 **XXXX**" | 用简单的否定句强行制造洞察假象。 |
| **定义的傲慢** | "**XXX** 的本质是 **XXXX**" | 还没讲事实，就急着给事物盖棺定论。 |
| **爹味儿劝诫** | "别被 **XXXX** 唬住/迷了眼，核心是..." | 预设读者是小白，自己是上帝视角。 |
| **廉价的真诚** | "我得说一句实在话 / 说白了" | 这种垫场话往往后面接的是更大的废话。 |
| **套路化互动** | "怎么办？为什么？我直接回三条！" | 强行制造节奏感，像是设定好的程序。 |

### B. 推荐：硬核工程师句式

#### 1. 描述事实（用"观察"代替"洞察"）
- "今天调了一下午，发现了一个细节：……"
- "从数据/代码表现来看，情况是这样的：……"
- "这事儿没那么玄学，主要就是因为……"

#### 2. 展示过程（用"尝试"代替"定论"）
- "原本以为是 A 的问题，后来发现坑在 B 身上。"
- "我们试了三种方案，最后只有这个暴力逻辑跑通了。"
- "绕了一圈回来，发现最管用的还是这几行代码。"

#### 3. 设定边界（用"局部性"代替"绝对化"）
- "在 X 这个特定配置下，会有明显的 Y 现象。"
- "我不确定这是否普适，但在我这儿，它解决了 Z。"
- "目前的方案上限也就到这了，再往后就是浪费算力。"

#### 4. 高阶递进（不仅是……关键是……）
- **逻辑**：前句讲"显性功能"，后句讲"隐性价值/工程痛点"。
- **例子**："不仅是代码写完了，关键是它把以前那些冗余的依赖链全理顺了。"

### C. 去油三技巧

1. **动词 > 名词**：别写"进行性能的深度优化"，写"把算子利用率拉到了 90%"。别写"认知的跨越"，写"以前那一套可以扔了"。
2. **具体 > 抽象**：别写"底层逻辑"，写"显存分布"或"推理延迟"。别写"维度"，写"应用场景"或"硬件约束"。
3. **Bitter Lesson 视角**：少吹捧"人类精妙的设计模式"，多关注"通用算法和大规模计算"。承认复杂性——如果 50 万行代码就是一团乱麻，直接说它"线头太多，抓不住"，而不是"蕴含深刻的设计哲学"。

### D. 固定句式库 (Sentence Stencil Library)

以下句式是 Why.J 文风的"骨架词"，可以直接套用。每条都标注了使用场景。

#### 开头 / 引入
- `这事儿没那么玄学，主要就是因为 [具体原因]。` — 打破迷思，快速进入事实。
- `今天调了一下午，发现了一个细节：[具体发现]。` — 用个人经历做钩子，带出技术细节。
- `最近在搞 [X]，遇到一个很典型的坑：[Y]。` — 工程实战开场白，直接进入战场。
- `从 [数据/代码/日志] 来看，情况是这样的：[事实描述]。` — 数据先行，不带预设立场。

#### 过渡 / 转折
- `原本以为是 [A] 的问题，后来发现坑在 [B] 身上。` — 展示调试过程，制造叙事张力。
- `绕了一圈回来，发现最管用的还是 [最简单的方案]。` — Bitter Lesson 式的自嘲，比强行升华更有说服力。
- `我们试了 [N] 种方案，最后只有 [暴力/简单方案] 跑通了。` — 用数字说话，承认过程的曲折。
- `说到 [X]，很多人会想到 [Y]。但实际场景里，[Z] 才是真正的瓶颈。` — 纠正常见误解，但不用"不是…而是…"的油腻句式。

#### 论点 / 判断
- `在 [具体条件] 下，会有明显的 [现象]。` — 限定边界，不搞绝对化。
- `我不确定这是否普适，但在我这儿，它解决了 [具体问题]。` — 承认局限性的诚实。
- `目前的方案上限也就到这了，再往后就是浪费 [算力/时间]。` — 对天花板的清醒认知。
- `这东西不是不好，而是 [不适合当前场景] 。` — 区分"工具本身"和"工具的适用性"。

#### 递进 / 总结
- `不仅是 [显性成果]，关键是它把 [隐性痛点] 全理顺了。` — 显性 + 隐性的双层次总结。
- `好处是 [A]，代价是 [B]。值不值，看你自己的场景。` — 把判断权还给读者。
- `简单说：[一句话结论]。展开说：[后续段落]。` — 信息密度控制，让读者自行选择阅读深度。
- `代码就这几行，但背后踩的坑，值得单独写一篇。` — 用轻描淡写掩盖过程的艰辛，效果比详细铺陈更好。

#### 收尾 / 留白
- `以上就是我目前的理解。如果有更好的做法，欢迎交流。` — 开放式结尾，不装全知。
- `先这样。等跑起来再说。` — 极简收尾，暗示这只是开始。
- `这事儿还有得琢磨。` — 承认未完成，比强行给结论更显专业。

#### 自嘲 / 真诚（来自李沐、田渊栋）
- `如果今天再来回答这个问题，我会说："我就是脑子抽了"。` — 用自嘲化解严肃，比强行升华更有力量。（李沐）
- `没了团队，我可能得转行去卖课了（此处不需要掌声）。` — 括号里的补充是活人感的精髓——像说话一样写。（李沐）
- `与其到处抱怨世道不公，不如就在大家面前装成自己刚刚升职吧，结果半年后果然升了职。` — 用反讽讲真实经历，信息量藏在语气里。（田渊栋）
- `所谓"仕途不幸诗家幸，赋到沧桑句便工"，生活太平淡，人生就不一定有乐趣了。` — 用古诗词消解沉重，比直接喊"我要坚强"高级得多。（田渊栋）
- `我的优点是听劝，真就去做LLM了。` — 三个字"真就"比"经过深思熟虑后决定"有说服力一百倍。（李沐）

#### 过程还原 / 现场感（来自李沐、田渊栋、赵程阳）
- `灵机一动，直接给老黄写邮件。老黄秒回说他来看下。一个小时后超微的CEO就打电话过来了。` — 用精确的时间线和动作还原决策现场，读者像在看纪录片。（李沐）
- `螃蟹吃到怀疑人生，遇到了各种匪夷所思的bug。` — "吃到怀疑人生"是口语化的极限表达，比"遇到了严峻挑战"有画面感。（李沐）
- `对字里行间的痛苦，很是共情。` — 不展开细节，用"共情"两个字传递重量，克制而有力。（李沐）
- `最近一两年的说话和做事方式，都是抱着一种"公司快把我开了吧"的心态，反而越来越放开。` — 用引号框住内心独白，让读者直接听到声音。（田渊栋）
- `虽然在那时，因为来自上面的巨大压力，不同意是几乎不可能的。` — 用括号式的补充说明交代背景约束，不隐瞒局限。（田渊栋）

#### 认知递进 / 阶段总结（来自李沐）
- `对LLM的认知经历了四个阶段。第一阶段是……第二阶段是……` — 用"阶段"而非"洞察"来组织叙事，暗示认知是演化的，不是顿悟的。（李沐）
- `第三阶段的认知是对于具体应用，我们是可以打赢市面最好模型的。` — "认知是"三个字直接给结论，不加修饰，因为事实本身够硬。（李沐）
- `那第五阶段认识呢？现在仍在进行中，希望能很快分享。` — 留白式结尾，承认未完成，给读者期待感。（李沐）

以下作者/平台的写作风格是 Why.J 文风的**直接参照物**。不是学习他们的选题，而是学习他们”怎么说话”——朴实、锋利、不端着。

- **田渊栋 ([2025年年终总结](https://zhuanlan.zhihu.com/p/1990809161458540818))**: **理性叙事者**。被裁、被边缘化这些沉重话题，用回报矩阵和概率论拆开来讲，不带情绪但带温度。”仕途不幸诗家幸，赋到沧桑句便工”，把人生波折化为创作素材的从容，是真正的智性克制。写可解释性研究时，用”第谷—开普勒—牛顿”的历史类比讲清楚一个领域的成熟路径，不堆术语，让非专业读者也能跟上逻辑链。是”具体 > 抽象”和”设定边界”的标杆。

## 7. 风格宣言 (Style Manifesto)
...

- **反营销**: 严格避免使用 "game-changing", "revolutionary", "paradigm shift" 等词汇。
- **视觉**: 相比于图库照片，更倾向于使用 **Geek Stick Figures (极客火柴人)** 或结构化的 Mermaid 图表。
- **数据胜过形容词**: 用具体的延迟数值/P99 代替“非常快”。
- **基底**: 将文本视为逻辑的**结构化基底 (Substrate)**，而不仅仅是交流媒介。
