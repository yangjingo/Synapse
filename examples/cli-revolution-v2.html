03 / Efficiency

维度：为什么 CLI 胜过 MCP / GUI？

  • 母语: LLM 在海量源码与文档中练就了极强的 CLI 直觉。
  • 效率: CLI 实现 O(1) Context，省去数万 Token 的 Schema 开销。
  • 架构: 纯文本输入输出，零模态转换，彻底消除截屏识别幻觉。
  • 组合: Unix 管道哲学天然适配 Agent 的多步逻辑编排流。 
$ cat logs.txt | grep "ERROR" | awk '{print $NF}'
# 极简命令实现复杂过滤，Token 损耗近乎为零
「 简洁是 Token 效率的唯一救赎。 」
whyj + nano banano + 2026/03/30
04 / Evolution

范式转移：Code-as-Action 的崛起

  • 传统: 预定义静态 Tool 列表，不仅昂贵且容易引发幻觉。
  • 新范式: 模型直接编写动态脚本（Bash/Python）来发现并操作。
  • 确定性: 减少中间层干扰后，Agent 的任务完成率可显著提升。
  • 本质: CLI 让 Agent 从“猜接口”进化为“做工程”，回归逻辑本质。
AGENTIC LOOP:
Reason → Code (CLI) → Observation → Reason
「 复杂的工具列表是 Agent 的杂讯。 」

05 / Product

Built for AI Agents: 飞书与钉钉的实践

  • Lark CLI: 深度优化的 Skills 架构，直接服务于 Agent 自动化逻辑。
  • DWS CLI: 动态发现架构，拒绝硬编码，适配 AI 的探索式交互。
  • 自愈机制: 提供结构化报错，引导 Agent 实时自动修复参数。
  • 审慎设计: 内置 --dry-run 预演模式，注入工程安全基因。

06 / Evaluation

确定性验证：从 Vibe-check 转向 State-check

  • 难题: 传统的 Chat 模式只能靠“感觉”好不好来判断。
  • Terminal-Bench: 通过 Exit Code 和 File State 进行硬核验证。
  • 闭环: 只有 CLI 能提供 Agent 逻辑闭环所需的“非黑即白”的反馈。
  • 指标: 成功率（Pass@1）是衡量 Agent 核心推理能力的唯一硬通货。
VERIFICATION ORACLE:
  • 1. Env Setup (Docker)
  • 2. Agent Action (CLI)
  • 3. State Check (Binary)
